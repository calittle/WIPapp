Documaker WIP edit

This web application is a customized implementation of WIP edit functionality for customer usage

Support

This application is not supported by Oracle Support. Underlying functionality of the web application, such as requests to Documaker Web Services and Docupresentment are supported, however requests to Support for this application will require you to engage Oracle Consultinbg.

The application requires the following components:

  • Oracle Documaker Web Services 12.x (Schema version V0)
  • Oracle Documaker Docupresentment, requires IDMK request types
  • Oracle WebLogic 10.3.x + (should work with 11g/12c releases)
  • Java 1.6+ (tested with 1.6 only, but should be forward compatible)
  • WIPedit Plugin-compatible browsers (currently IE and Firefox only).
  • A configured WebLogic security realm that includes DOC_PREP and DOC_VET groups, which are then linked to this application's definition for those roles.
  • End users will require the appropriate version of the Documaker WIP edit plugin to be installed prior to using this application.

Optional components/configuration:

  • SSL configuration. Note that some browsers may require specific certificate configurations to enable SSL
  • SSO configuration. This is done strictly in WebLogic types
  • Oracle WebLogic 10.3.x + (should work with 11g/12c releases)
  • Java 1.6+ (tested with 1.6 only, but should be forward compatible)

Application Parameters

The following settings are kept in the WEB-INF/web.xml file and may be modified to suit your implementation. Note: there are settings other than these that are not listed here. These settings should not be changed.

Parameter Default Value Description
hostUrlPrefix http://192.168.1.72:7001 The Server/hostname and port where WIPEDIT will be deployed.
pegaEndpoint no default [currently not used] The endpoint for the PEGA web service to update on transaction change.
idsConfig CORRESPONDENCE The name of the IDS configuration used for all requests and MRL components. Verify this setting in a few places:
[ODEE_HOME]/documaker/docupresentment/dap.ini - [ Configurations ] Config= Correspondence
[ODEE_HOME/documaker/mstrres/[MRL_DIR e.g. dmres]/fsisys.ini - [ CONFIG:Correspondence ]
logLevel ERROR Values can be [DEBUG,ERROR,WARN]. Controls logging level output to managed server .out file. Use DEBUG for trace level output.
docPrepGroupId 1021 The Entity ID of the DOC_PREP group (check the DMKR_ENTITIES table for the ID value to use here).
docVetGroupId 1022 The Entity ID of the DOC_VET group (check the DMKR_ENTITIES table for the ID value to use here).
docVetGroupName DOC_VET The name of the group representing Document Reviewers.
docPrepGroupName DOC_PREP The name of the group representing Document Preparers.

Security Parameters

The following settings are kept in the WEB-INF/web.xml file and may be modified to suit your implementation. These settings are specific to security implementation within J2EE applications and are not specific to WebLogic or the WIPedit implementation.

Parameter Default Value Description
<user-data-constraint><transport-guarantee> NONE Controls usage of SSL. Valid values are NONE, INTEGRAL, CONFIDENTIAL.
None = no security, messages are not encrypted or secured.
INTEGRAL = message contents are hashed to prevent tampering, but are reading.
CONFIDENTIAL = messages are encrypted and hashed to prevent tampering and reading.

Use CONFIDENTIAL secure implementations.
<login-config><auth-method> BASIC Determines how users are authenticated.
BASIC - challenge/response in plaintext.
DIGEST - challenge/response encrypted.
CLIENT-CERT - SSO/certificate usage.

Use CLIENT-CERT for SSO.
<login-config><realm-name> myrealm The name of the WebLogic security realm for the domain.

WebLogic Settings

The following settings are kept in the WEB-INF/weblogic.xml file and may be modified to suit your implementation. These settings are specific to security roles. Make sure you have already configured the security realm within WebLogic to map users/groups to these roles.

Parameter Default Value Description
<role-name>DocumentPreparer</role-name> <principal-name>DOC_PREP</principal-name> DOC_PREP This maps the role DocumentPreparer to the WebLogic group DOC_PREP. If you use a different WebLogic group name, you can change it here. Do not change the role-name!
<role-name>DocumentReviewer</role-name> <principal-name>DOC_VET</principal-name> DOC_VET This maps the role DocumentReviewer to the WebLogic group DOC_VET. If you use a different WebLogic group name, you can change it here. Do not change the role-name!
There are other security-role-assignment values in this file. Do not change them.

Implementation

Adjust settings as necessary in the WAR file. Depending on the tools you have available, you may be able to directly edit the settings in the various files, skipping the file unzipping/zipping steps.

  1. Unzip the WAR file into a temporary directory [temp_dir] .
  2. Enter the [temp_dir] and navigate to /WEB-INF/web.xml file.
  3. Update application settings according to your environment. Save the file changes. Take care not to update settings that are not listed in this file. They are not meant for you to change.
  4. Navigate to /WEB-INF/weblogic.xml file.
  5. Update WebLogic application settings according to your environment. Save the file changes.
  6. Update Integration endpoint to Documaker Web Services.
    • Navigate to WEB-INF/classes/oracle/dws
    • Open CompositionService.wsdl for editing with a text editor.
    • Near the bottom of the file, locate the entry 
      <soap:address location="http://192.168.1.72:7001/DWSV0AL1/CompositionServiceSoap12"/>
    • Update the file with the proper server, port, and endpoint for your implementation. Ideally you should only change the server/port values within the URL. You may need to change the web service application if you are integrating with a specific assembly line or schema version. If you have any doubts, you can refer to your WebLogic console for valid values.
    • Save the file.
  7. Select all the contents of the files inside the [temp_dir] and zip into a file
  8. Rename the file with a WAR extension, e.g. "wipedit.war"
  9. Proceed to deployment .

To deploy, copy the updated WAR file to your WebLogic application server. Place the file in the same place each time, that way if you have to update it later, you can follow the same procedure. We recommend placing this file in the ODEE_HOME/documaker/j2ee/weblogic/oracle11g/ directory.

  1. Open WebLogic console for the idocumaker_domain domain and login.
  2. In the Domain Structure click Deployments
  3. Click the Install button.
  4. Navigate to the directory where you deposited the WAR file.
  5. Click the radio button to select wipedit.war and then click Next .
  6. Select "Install this deployment as an application" and then click Next .
  7. Select the target for the application (recommended server is "dmkr_server") then click Next .
  8. Use the default settings for "DD Only" and "Use the defaults defined by the deplyoyment's targets, then click Finish .

Usage

To integrate this application with your existing applications, you need only to utilize the following URL when you want to invoke the application:

http(s)://[ server:port ]/wipedit?uniqueid=[ uniqueId ]&taskid=[ taskId ]

Where:

  • [ server:port ] is the hostname and port where the WebLogic application WebLogic can be reached. If you are using an HTTP frontend that has the WebLogic plugin, you can configure the plugin to forward requests as necessary. Such configuration is beyond the detail of this document.
  • [ uniqueId ] is the unique identifier of the transaction you wish to edit.
  • [ taskId ] is the task ID (aka docId) of the transaction you wish to edit. Note: this value is not actually used, but is required.

Note: HTTPS can be used if SSL is configured. Such configuration is supported by WebLogic and this application, but configuration thereof is outside the scope of this document.

For end users, operation is simple. There are three functions:

  • Save - this saves any changes made. This should not be used by Document Reviewers.
  • Submit - this marks the transaction's review or edit operation complete and unlocks the transaction. This does not save changes!
  • Close - this closes the WIPedit window. This does not perform a Save or Submit!
